Why Feature-Based ArchitectureThe previous plan used a component-type structure — every page in /pages, every API call in /api, every store in /store. Working on inventory meant jumping between 5 different folders constantly. Feature-based fixes this.Component-Type Structure (v2.0)Feature-Based Structure (v3.0)src/pages/InventoryPage.jsxsrc/features/inventory/InventoryPage.jsxsrc/api/products.api.jssrc/features/inventory/inventory.api.jssrc/store/productStore.jssrc/features/inventory/inventoryStore.jssrc/components/forms/ProductForm.jsxsrc/features/inventory/ProductForm.jsx5 folders open to work on one feature1 folder open to work on one featureThe Feature-Based Rule — Memorise This•  A file belongs to a feature if only that feature uses it — it lives inside that feature's folder•  A file moves to shared/ the moment a second feature needs it — not before•  shared/ contains: UI primitives (Button, Input, Toast), layout (Sidebar, Header), and cross-feature hooks•  Features never import from each other — they only import from shared/•  Adding Phase 2 (billing) means adding features/billing/ — nothing else in the tree changes•  Deleting a feature means deleting one folder — no hunting across the codebase2. Import Rules Between FeaturesFollow these rules to prevent circular dependencies and keep features isolated.Allowed ImportExampleWhyfeature → shared/uiinventory imports Button, DataTableShared UI is designed to be imported everywherefeature → shared/storedashboard imports authStoreAuth state needed by all featuresfeature → shared/hooksinventory imports useDebounceGeneric hooks belong in sharedfeature → shared/layoutpages use AppLayoutLayout wraps all pagesfeature → shared/api/axiosall feature api files import axios instanceAxios config is sharedpricing → inventoryStorepricing reads product cachePricing only reads, never writes to inventory storeeod → inventoryStoreeod reads product list for searchEOD only reads, never writes to inventory storedashboard → StockAdjustDrawerdashboard imports from features/inventoryThis is acceptable — dashboard only renders itNever Do These•  NEVER: features importing from each other's API files (e.g. pricing importing inventory.api.js) — use the shared axios instance directly or move to shared/api/•  NEVER: Two features sharing a Zustand store that both write to — only one feature owns a store, others read only•  NEVER: Putting a component in shared/ before a second feature needs it — wait for the second use case•  NEVER: Business logic in shared/ui components — Button should never know what a product is•  NEVER: Importing from a sibling page (InventoryPage importing from ProductFormPage) — communicate via router or store only7.
